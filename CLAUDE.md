## Architecture

### Static Export on Cloudflare Pages

This is a **Next.js 15 App Router** project built with `output: 'export'` — it produces a fully static `out/` directory deployed to Cloudflare Pages. Under static export these features do **not** work and will break the build:
- API routes, middleware, `next/headers`, `next/server` runtime
- `export const dynamic = 'force-dynamic'` or `export const revalidate`
- Server actions, cookies(), headers()

All data must be available at build time. If you need server features, migrate the deployment model first. Run `npm run validate:static-export` before any build-affecting change.

**Security headers** are defined in `public/_headers` (Cloudflare Pages format) and validated by `scripts/ci/validate-security-headers.mjs`. Next.js `headers()` config is unsupported under static export and is not used.

### Data Pipeline (workbook → `public/data`)

The sole authoritative data source is `data-sources/herb_monograph_master.xlsx`. All runtime JSON in `public/data/**` is generated and **disposable** — regenerated by `npm run data:build`.

The pipeline runs these steps in order:
1. Validate workbook source (`validate-workbook-source.mjs`)
2. Parse workbook → `herbs.json`, `compounds.json`, detail files (`build-runtime-from-workbook.mjs`)
3. Validate sleep evidence engine
4. Post-process workbook payloads
5. Apply governance overlay
6. Build related runtime maps (`herb-compound-map.json`, etc.)
7. Build summary indexes
8. Generate freshness metadata
9. Build route manifest
10. Build internal link engine
11. Build sitemap manifest
12. Build export batches
13. Build semantic snapshots
14. Build search index

**Editing data:** For broad or structured changes, edit the workbook then run `npm run data:build` so edits survive regeneration. For quick fixes you can edit `public/data` JSON directly — the next `data:build` will overwrite it unless the workbook is also updated. See `docs/workbook-only-data-contract.md`.

### Active vs. Legacy Code (`src/`)

Active code paths:
- `app/`, `components/`, `lib/` (root-level) — production App Router, type-checked, a11y-strict
- `src/lib/runtime-*.ts`, `src/lib/cloudflare-image-loader.ts` — active runtime helpers
- `src/components/explore/`, `src/components/runtime/`, `src/components/Footer.tsx`, `src/components/mobile-bottom-nav.tsx`
- `src/store/` — Zustand state (client-side only, no server state under static export)

Legacy/excluded paths (see `tsconfig.json` excludes):
- `src/pages/**/*` — old pages router, excluded from type-checking
- Most of `src/components/**/*` not listed above

`@/*` path alias resolves to `./` (root-first), so active code under `app/`, `components/`, `lib/` is preferred over `src/` equivalents.

### `lib/` vs `src/lib/`

`lib/` (root-level) is active production logic. `src/lib/` is legacy but some files there (`cloudflare-image-loader.ts`, `runtime-*.ts`, `seo.ts`) are still imported. Check `tsconfig.json` paths and `vitest.config.ts` aliases to understand which `@/lib/*` imports resolve where.

### Agent Enrichment System (`agent/`)

A separate pipeline that orchestrates AI-assisted content enrichment. Agents produce patch files validated in CI before being applied to the workbook.

**Workflow:**
1. Run agents: `npm run agent:run --mode=standard --batch=5`
2. Patches appear in `agent/patches/{date}/*.json`
3. CI validates patches automatically (`npm run validate:agent-patches`, part of `check:full`)
4. Review: `npm run agent:review` → creates `ops/agent-review/approved-patches.{json,csv}`
5. Manually merge approved data into the workbook
6. Run `npm run data:build` to regenerate `public/data`

Patches do **not** modify `public/data` directly. See `docs/agent-integration-guide.md`.

## Theme and Styling

- Light and dark mode supported via `DarkModeProvider`/`DarkModeToggle` (header toggle). Use `dark:` classes on new pages; keep light and dark variants in sync.
- CSS variables in `app/globals.css`: `--bg: #fffdf7` (warm off-white), emerald accent `#358f52`, plus `.dark` overrides.
- Fonts: Inter (body) + Fraunces (display), self-hosted via `@fontsource-variable` packages (not `next/font/google`).
- **Tailwind CSS v4** — no `tailwind.config.js`; all customization is via CSS custom properties.
- No `tailwind.config.js` — customization via CSS variables in `app/globals.css` only.

## Build Configuration Notes

- `trailingSlash: true` in `next.config.mjs` — canonical URLs always have trailing slashes
- `cpus: 2` experimental limit — prevents OOM on memory-constrained CI hosts
- Custom image loader: `src/lib/cloudflare-image-loader.ts`; Amazon CDN remote patterns are allowed
- `__BUILD_DATE__`, `__BUILD_TIME__`, `__COMMIT_HASH__`, `__APP_VERSION__` injected at build time via Webpack DefinePlugin
- Blog/articles use `content-collections` — run `npm run content:build` (called automatically by `pretypecheck`)
