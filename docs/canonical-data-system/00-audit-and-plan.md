# Canonical Data System — Audit & Implementation Plan

> Status: **Planning pass (Prompt 1)**. No code changed in this pass.
> Author: automated audit. Date: 2026-07-11.

This document is the output of a full-repository audit of how The Hippie Scientist
site sources, transforms, and consumes its herb/compound data, plus a concrete
implementation plan for migrating to a Git-friendly canonical data system with a
generated SQLite database, an explicit semantic graph, and a multi-GPT patch inbox —
**without** a hosted database and **without** breaking the existing Next.js static
export on Cloudflare Pages.

---

## Part 1 — Audit findings

### 1.1 Where the workbook / exported data lives

| Location | Role |
|---|---|
| `data-sources/herb_monograph_master.xlsx` | **Sole authoritative source of truth** today. A 22-sheet Excel workbook. |
| `public/data/**` | **Generated, disposable** runtime JSON (~50 top-level files + subdirectories). Regenerated by `npm run data:build`. |
| `scripts/workbook-source.mjs` | Resolves and asserts the workbook path (env override `HERB_XLSX_PATH`). |
| `scripts/utils/read-workbook-exceljs.mjs` + `scripts/data/workbook-parser.mjs` | ExcelJS-based reader (trusted local build only). `xlsx` package is intentionally **not** installed; ExcelJS 4.4.0 is used. |

The workbook read path has a hardened fallback: full read → namespace-normalized retry → streaming row reader. It works (all 22 sheets parse).

### 1.3 Site pages / components that consume the data

Runtime data is read from `public/data/**` (never from the workbook at runtime — that's a hard invariant, `verify-workbook-only-path.mjs`). Key consumers:

- `lib/server/runtime-data.ts` — `getAllHerbs()`, `getAllCompounds()`, `getHerbBySlug()`, etc. Reads `herbs.json`, `compounds.json`.
- `src/lib/runtime-data.ts`, `src/lib/runtime-summary-indexes.ts`, `src/lib/runtime-related-maps.ts`, `src/lib/herb-data.ts`, `src/lib/runtime-record-index.ts` — evidence engine, summaries, related maps.
- `app/herbs/[slug]/`, `app/compounds/[slug]/`, `app/goals/[slug]/`, `app/stacks/[slug]/`, `app/compare/[slug]/` — depth-layer routes.
- `app/sitemap.ts`, search (`lib/search/*`, `app/search/SearchClient.tsx` → `search-index.json`), semantic linking (`lib/semantic-runtime.ts`, `lib/graph-query.ts`, `lib/graph-links.ts`), comparisons (`lib/compare.ts`), freshness (`lib/freshness.ts`).

The site expects, at minimum, arrays of records with the ~40 keys shown for `herbs.json` (`slug`, `name`, `summary`, `description`, `effects`, `mechanisms`, `canonical_mechanisms`, `evidence_grade`, `contraindications`, `interactions`, `side_effects`, `dosage`, `forms`, `conditions`, `tags`, affiliate fields, `last_updated`, …), plus derived index/map/snapshot files.

### 1.4 Is data duplicated across formats?

Yes, extensively — but **by design** (workbook = source, everything else = generated). Duplication exists in:

- **JSON** — `public/data/**` (~50 files: `herbs.json`, `compounds.json`, detail dirs, indexes, maps, snapshots, search index, graph).
- **TypeScript** — `data/*.ts` and `lib/*.ts` hold curated/editorial content (`data/goals.ts`, `data/comparisons.ts`, `data/best.ts`, `lib/goal-clusters.ts`, `lib/citations.ts`, etc.). Some of this is **hand-authored and NOT derived from the workbook** — it is a second, parallel source of truth for goal/compare/editorial layers.
- **JSONL** — `public/data/enrichment-*.jsonl` (agent enrichment).
- **Markdown/MDX** — `content/`, blog/articles via `content-collections`.

The risk area is the **TypeScript curated layer** — it is real source data that lives outside the workbook and outside any schema/validation.

### 1.5 How entities/claims/edges are currently linked

- **Entities** keyed by `slug` (herbs + compounds share the `slug` namespace via `Entity_Master`).
- **Herb→compound** via `Entity_Relationships` (`contains_compound`, `direct herb-compound map`) → generated `herb-compound-map.json`, `herb-compound-links` in `lib/`.
- **Mechanisms/pathways** linked by canonical-mechanism normalization (`Taxonomy_Rules` → `canonical-mechanisms.json`, `pathways.json`), shared-mechanism overlap edges (`pathway-overlap`, `mechanism-overlap`).
- **Claims** (`Evidence_Register`) linked to entities by `entity_slug` and to sources by `pmid`/`doi`/`url` → `claims.json`, `public/data/evidence-engine/`.
- **Sources** (`Source_Register`) linked by `entity_slugs` / `related_ids` / `pmid` → `publication-*.json`.
- Linkage is **slug-string based**, not ID based. There is no stable non-slug entity identifier — renaming a slug breaks joins (mitigated by `Slug_Alias_Map`).

### 1.6 Fields that look incomplete / inconsistent / hard to maintain

- **Slug-as-primary-key** — brittle; alias map is a patch over a missing stable ID. Conflicting slugs already tracked in `Slug_Alias_Map`.
- **Herb/compound in one 52-col sheet** — many columns only apply to one type; sparse.
- **Governance filtering drops records** — `Entity_Master` has 293 herbs / 588 compounds but runtime emits **291 herbs / 565 compounds** (restricted-term + status filtering in `build-runtime-from-workbook.mjs`). The delta must be explained and preserved, not lost.
- **Legacy sheet-name references** in the build script (V3 names) — dead aliases, confusing.
- **Editorial TS layer** (`data/*.ts`) — unschematized parallel data.
- **Evidence grade vs. tier vs. confidence tier** — three overlapping evidence dimensions; inconsistent population across rows.
- **Delimiter/format drift** in multi-value cells (effects, tags, aliases) — pipe/comma/semicolon mixed; whitespace/casing variance.
- **Provenance is implicit** — `source_table`/`source_row`/`import_note` exist on some sheets but not uniformly.

### 1.7 Build scripts that transform / validate the data

Pipeline (`data:build`, 14 steps): `validate-workbook-source` → `build-runtime-from-workbook` → `validate-sleep-evidence-engine` → `postprocess-workbook-payloads` → `apply-governance-overlay` → `build-related-runtime-maps` → `build-runtime-summary-indexes` → `generate-freshness-metadata` → `build-route-manifest` → `build-internal-link-engine` → `build-sitemap-manifest` → `build-export-batches` → `build-semantic-snapshots` → `build-search-index`.

Validators: `validate-workbook-source`, `validate-workbook-schema`, `validate-workbook-json-parity`, `verify-workbook-only-path`, `guard-no-full-build-drift`, `validate-data-next`, `semantic-governance-check`, `validate-indexability-metadata`, `guard-generated-data`, `verify-generated-data`, plus `data:audit` (`audit-source-of-truth`, `audit-workbook-gaps`).

Existing enrichment/patch surface: `schemas/patch.schema.json`, `agent/` orchestrator, `agent/patches/{date}/*.json`, `scripts/review-patches.mjs`, `public/data/enrichment-*.jsonl`. **This is the natural integration point for the new GPT patch inbox** — the new system should generalize it, not fork it.

### 1.8 What breaks if the workbook is replaced as the direct source of truth

These guards **assume the workbook is the only source** and will fail if canonical JSONL becomes the source without adaptation:

- `validate-workbook-source.mjs`, `validate-workbook-schema.mjs` — require the xlsx to exist and match a schema.
- `verify-workbook-only-path.mjs` — asserts runtime is generated **only** from the workbook.
- `validate-workbook-json-parity.mjs` — asserts `public/data` matches a fresh workbook parse.
- `guard-no-full-build-drift.mjs`, `guard:source-of-truth` — drift guards keyed to the workbook.
- `data:build*` scripts — all read the workbook directly.

**Strategy:** keep the workbook path fully working as-is; add the canonical system **beside** it; flip the "source of truth" guards only at Phase 6, behind a comparison report proving zero unexplained drift. The workbook becomes a *migration input / archival snapshot*, not the live source.

## Part 2 — Implementation plan

### 2.1 Recommended folder structure

```
data/
  canonical/
    entities/           # <type>.jsonl  (herb, compound, effect, condition, mechanism, study, source, safety-issue, interaction, preparation)
    claims/             # claims.jsonl
    relationships/      # edges.jsonl
    sources/            # sources.jsonl
  patches/
    inbox/              # raw incoming GPT patches (any supported format) — preserved permanently
    normalized/         # canonical normalized patch JSON
    applied/            # successfully applied patches + result metadata
    rejected/           # rejected patches + reason
  generated/            # graph exports, reports, sample site data (disposable)
  staging/              # migration output before promotion to canonical/ (disposable)
  schema/               # JSON Schema mirrors of the Zod schemas (for external GPT authors + CI)
  audit/                # immutable audit-log entries (append-only .jsonl)
  snapshots/            # timestamped pre-apply snapshots for rollback
  db/                   # generated canonical.sqlite (disposable, gitignored)

scripts/data/           # (existing dir) new scripts added here
lib/canonical/          # shared TS: Zod schemas, ID hashing, JSONL io, field-alias config loader
config/
  field-aliases.json    # configurable patch field-alias map
docs/canonical-data-system/   # this doc + generated reports + daily-use/troubleshooting/recovery guides
```

`data/db/`, `data/generated/`, `data/staging/` are gitignored. `data/canonical/**`, `data/patches/**` (raw inbox + applied/rejected records), `data/audit/**`, `data/schema/**` are committed.

### 2.2 Proposed entity schema (Zod)

Common envelope for **every** entity:

```
id            string   # stable deterministic — "ent_<type>_<hash12>" from (type + canonical seed)
entity_type   enum     herb | compound | effect | condition | mechanism | study | source | safety_issue | drug_interaction | preparation
canonical_name string
slug          string
aliases       string[]
description    string
review_status enum     approved | pending | needs_review | deprecated
created_at    ISO datetime
updated_at    ISO datetime
provenance    { source: string, source_ref?: string, migrated_from?: {sheet,row}, notes?: string }[]
legacy        record   # unmapped/legacy fields preserved verbatim (never discard)
```

Type-specific fields live in a `data` sub-object validated by a per-type refinement (e.g. herb: `latin_name`, `effects`, `mechanisms`, `evidence_grade`, `dosage`, `forms`, `contraindications`, governance flags; study: `pmid`, `doi`, `year`, `design`, `population`, `sample_size`).

### 2.3 Proposed claim schema

```
id            string   # "clm_<hash12>" from (subject + predicate + object + qualifiers)
subject_id    string   # entity id
predicate     enum     affects_mechanism | supports_outcome | has_dosage | has_safety_warning | contraindicated_in | interacts_with | contains | ...
object_id     string?  # entity id  (xor object_literal)
object_literal any?     # for dosage ranges, populations, free text
qualifiers    record   # population, dose, duration, direction, claim_tier, ...
source_ids    string[]
evidence_level enum     human_rct | human_obs | preclinical | traditional | anecdotal | none
confidence    number   # 0..1
review_status enum
notes         string
created_at / updated_at / provenance / legacy
```

### 2.4 Proposed relationship (edge) schema

```
id            string   # "edg_<hash12>" from (from + type + to)
from_id       string
rel_type      enum     contains_compound | shares_mechanism | shares_compound | affects_effect | targets_condition | interacts_with | pathway_overlap | ...
to_id         string
direction     enum     directed | undirected
weight        number?
confidence    number?
source_ids    string[]
evidence_level enum
origin        enum     explicit | derived | suggested   # provenance class for graph honesty
explanation   string?  # REQUIRED when origin=suggested
review_status enum
provenance / created_at / updated_at
```

### 2.5 Proposed patch schema (normalized)

```
patch_id            string
patch_version       string
created_at          ISO
generator           string?     # GPT/author
target              { id?, slug?, canonical_name?, alias?, entity_type }
operations          Operation[]
sources             Source[]
notes               string
confidence          number
requires_review     boolean
original_filename   string
original_hash       string      # sha256 of raw unmodified input
```

`Operation` union: `create_entity | update_field | add_alias | add_claim | update_claim | add_relationship | update_relationship | add_source | add_safety_warning | add_drug_interaction | deprecate | merge_candidates`.

Field-alias mapping lives in `config/field-aliases.json` (not hard-coded), e.g. `{ "canonical_name": ["common_name","name","herb_name","title"], "effects": ["benefits","uses","outcomes"], ... }`.

### 2.6 Migration steps (phased, mapped to Prompts 2–7)

1. **Foundation (Prompt 2)** — Zod schemas, JSONL io, ID hashing, `node:sqlite` build script, npm scripts (`data:validate`, `data:build-db`, `data:export`, `data:check`), tests. Empty canonical dirs. No workbook read yet.
2. **Migration (Prompt 3)** — `data:migrate-workbook` reads every sheet, maps rows → entities/claims/edges/sources, preserves sheet+row+col provenance, deterministic IDs, dry-run, writes to `data/staging/`, emits duplicate/missing/relationship/migration reports. Promote staging → canonical only after review.
3. **Patch normalization (Prompt 4)** — multi-format ingest → normalized patch, alias config, fixtures/tests.
4. **Patch application (Prompt 5)** — transactional apply, snapshots, rollback, audit log, idempotency, tests.
5. **Site integration (Prompt 6)** — export adapter canonical → exact current `public/data` shapes, deterministic ordering, staleness check, comparison report, keep workbook path as fallback until parity proven.
6. **Graph + finish (Prompt 7)** — SQLite views/indexes, query CLI, gaps/orphans/conflicts, graph export, CI, docs.

### 2.7 Risks & safeguards

| Risk | Safeguard |
|---|---|
| Silent data loss vs. workbook | Migration parity report; `legacy` field preserves every unmapped column; record-count reconciliation with explanations. |
| Governance-filtered records disappearing | Preserve ALL entities in canonical with governance flags; filtering happens only in the **export adapter**, matching current behavior exactly. |
| Route/slug changes → SEO breakage | Export adapter preserves slugs 1:1; comparison report diffs slugs/titles/meta/structured-data before switch; redirects unchanged. |
| Non-deterministic generated files | Deterministic sort + `SOURCE_DATE_EPOCH`-style fixed timestamps; staleness check compares content hashes. |
| Cloudflare has no DB/workbook at runtime | SQLite + workbook are **build-time only**; site reads generated JSON exactly as today. `node:sqlite` is built-in (no native dep). |
| GPT patches corrupting data | Dry-run default, review gates, never overwrite stronger sourced data, never delete on absence, snapshots + rollback, immutable audit log, raw patch preserved. |
| Two sources of truth during migration | Workbook path stays authoritative until Phase 6 comparison passes; canonical runs in parallel/shadow mode. |

### 2.9 How the site keeps working during migration

- The workbook → `public/data` pipeline is **untouched** through Phases 2–5. The site builds and deploys exactly as today.
- The canonical system is additive: new dirs, new scripts, new npm targets. Nothing in `app/`, `components/`, `lib/` runtime read paths changes until Phase 6.
- Phase 6 introduces the export adapter as an **alternative** producer of `public/data`, gated by a comparison report. Only after the report shows zero unexplained slug/title/meta/route loss does the build flip to `validate canonical → build SQLite → export site data → next build`.
- Rollback at any point = keep using the workbook pipeline; the canonical dirs are inert data.
